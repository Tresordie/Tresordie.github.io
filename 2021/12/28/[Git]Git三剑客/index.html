[Git]Git 三剑客

1. 创建本地仓库

  1. 命令创建本地仓库

    git init

  2. 默认创建了 master 分支, 由./git/HEAD 文件可以看出

  3. ./git/config 可以看出还没有做 user name 及 email 相关的配置

2. 最小配置

  1. 命令配置用户名及邮箱(git 可以发邮件, 如代码审核的邮件)

    git config --global user.name "your_user_name"

    git config --global user.email "your_email"

  2. config 配置的限定

    git config --global(全局于当前用户)

    git config --local(只对于某个仓库有效)

    git config --system(对系统中所有用户有效)

  3. 查看系统的配置

    git config --list

  4. 当 --local 和--global 同时设置后，--local 的优先级高于--global

3. 仓库增加内容到暂存区并提交

  1. 新增内容添加到暂存区

    git add xxx

  2. 将暂存区内容提交到版本目录

    git commit -m "更改原因" // commit 提交

    git commit -am "更改内容" // 不需要将更改添加到暂存区，直接进行提交

4. 本地创建的仓库与远端建立连接

https://www.cnblogs.com/mkl34367803/p/11220465.html

https://wychuan.gitee.io/git/set-upstream.html

  1. 本地文件夹同一个文件夹需要被 git 管理, 如工作文件夹已经添加了文件

    git init

    git add .

    git commit -m "marks"

  2. 本地工作区在进行 git init 之后，git 可以 track 更改了，但只是基于本地文件夹仓库，需要与 git 远端仓库发生关联

    git remote add origin 远端仓库链接地址 url

  3. 执行以上关联后，.git/config 中内容发生了变更, 里面多出了[remote:origin]部分, 只是添加了 origin 的指代(而没有指定是哪个分支，仓库可能有几个分支)

  4. 本地与远端关联后，本地如果修改文件，在 push 到远端之前需要先从远端 pull(fetch + merge), 然后才能 push

  5. 本都从远端拉取文件(最好以下面的命令 pull 远端并合并 local, 否则会出错 fatal: refusing to merge unrelated histories, 两个分支没有关联起来)

    git pull --rebase origin master

  6. 然后将 local 的更改 push 到远端

    git push origin master

5. git push --set-upstream 的作用

http://blog.sina.com.cn/s/blog_694270050102zbrq.html

6. 更改文件名的简便方法

  1. git mv 原文件名 更改后文件名
  2. git commit -m "change name"
  3. git push origin master

7. 清除暂存区及工作区所有更改

git reset --hard

8. 查看 git 之前提交的版本历史

  1. 单行 commit 格式查看

    git log --oneline // 一行一个 commit id 查看

  2. 查看最近几次的 commit

    git log --n4 --oneline // 查看最近 4 次的 commit 提交

  3. 查看所有的 git log 信息

    git log --all

  4. 以图表信息方式查看 commit

    git log --all --graph

  5. git log 的 help 相关菜单

    git help --web log

9. 基于某个 commit 创建 git 分支

  1. 基于某个 commit id 创建分支

    git checkout -b new_branch commitNum

  2. 更新变更

    git add .

    git commit "xxx"

    git push --set-upstream origin new_branch

  2. HEAD 文件内容

    • FETCH_HEAD 内容为当前最近的一个 commit number, 内容为 commit 的内容

  4. config 文件

    • config 文件里面都是一些配置文件，比如 core 信息，用户名&邮箱等等

11.分离头指针

  1. 什么是分离指针

    分离头指针是头指针指向的 commit 没有绑定某个分支

  2. 如何实现分离头指针

    git checkout commitnum

  3. 分离头指针在重新创建分支后，HEAD 可以重新指向新分支

13. commit 相关修改

https://blog.csdn.net/hu1628299958/article/details/109908395

  1. 修改最新一次的 commit message

    git commit -amend

  2. 修改老旧 commit message

    git rebase -i 被修改 commit 的父 commit

  3. 对于多个未提交的连续的 commit 进行合并

    git rebase -i 待合并的所有 commit 的父 commit

  4. 不连贯的多个 commit 进行合并

    git rebase -i 待合并的 commit 的父 commit

14. 差别比较

  1. 比较暂存区与 HEAD 所指文件的差异

    git diff --cached

  2. 比较工作区和暂存区之间有何差别

    git diff

  3. 不同分支之间差别比较

    git diff branch1 branch2

  4. 不同分支中指定文件的差别

    git diff branch1 branch2 --filename

  5. 不同 commit 中指定文件之间的差别

    git diff commit1 commit2 --filename

15. 恢复删除相关

  1. 暂存区恢复和 HEAD 一致

    git reset HEAD

  2. 消除最近的 commit 的提交

    git reset --hard commit

  3. 将工作区恢复到和暂存区一致

    git checkout --filename

  4. 取消暂存区部分文件的更改

    git reset HEAD --filename

  5. 删除仓库中的某个文件

    git rm filename

16. 临时需要在上一次提交上增加功能，本次更改需要被压栈保存

  1. 压栈保存

    git stash

  2. 查看压栈

    git stash list

  3. 弹栈恢复

    git stash apply // 弹栈后未删除 list 信息

    git stash pop // 弹栈后删除 list 信息

17. 多人协作过程中的注意点

  1. 提交 commit 并 push 之前，先 git pull 远端仓库最新 update 到本地

  2. 当协作的多人更改了同一个文件的同一个地方时，此时提交 commit 时产生的冲突需要和相关人员进行沟通确认最终提交

  3. 当协作的多人需要对同一个文件进行重命名，需要和相关人员沟通确定最终的重命名名称

  4. 多人协作时，禁止使用 push -f 进行强行 push; 当其他人员 push 了 commit, 此时我们没有及时的 pull, 但通过 push -f 会造成较新的一些提交被覆盖掉了

  5. 多人协作的项目，当大家都在主线进行开发时，禁止对主线进行变基操作，否则其他开发人员

    • 什么是変基. https://blog.csdn.net/weixin_39640298/article/details/85302187

    • 如何変基

      • 当有两个分支都是以一个基为父 commit
      • 两个分支各自进行 commit 提交
      • 切换到一个分支 git checkout branch2
      • 当前分支以另一个分支为基础进行変基 "git rebase branch1", 此时从 branch1 和 branch2 共有父 commit 开始，branch1 上的所有 commit 修改会添加到 branch2 中，并进行合并形成新的 commit, 最终之前的 branch1 主线变到了 branch2 分支了

    • 変基过程中的冲突

      • 変基过程中，如果有冲突，需要手动更改冲突部分
      • 重新添加进暂存区 git add -A
      • 继续 rebase; git rebase --continue
      • 然后 push 到远端 git push; 此时有可能无法识别当前是 branch1 还是 branch2, 系统会提示与远端的关联 git push --set-upstream origin branch2; 重新关联 push 就可以了

    • 変基带来的问题: 正常情况下，我们的想法是在同一条主线上进行开发，但是如果进行了変基，会造成一些 commit 无法会出现在了変基后的分支后，而之前的分支就会丢失部分 commit, 这就导致了部分开发的同学们的代码会出现某些 commit 不全的问题.
